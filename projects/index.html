Projects

Six pipelines I built and run. Each one gets the same treatment: the problem it exists to solve, the stages it actually runs, what broke and what I did about it, and the code that fixed it. The bugs are the interesting part, so they are not skipped. For the subsystems these run on, see Platform.

1. Automated server provisioning

The problem. A server is a fact recorded in five places at once: the address in IPAM, the machine on the hypervisor, the host in monitoring, the resource in infrastructure state, and the entry in configuration inventory. Build one by hand and any of those can silently disagree, or worse, silently agree with a machine that already exists. The failure this pipeline is built against is the one everybody has seen: a new host quietly reconciles a live one because nothing checked every system that already knew the name.

  1. Validate input
  2. Reachability pre-flight - refuse a target whose network path is untested
  3. Name collision check - ask every system that could already know the name
  4. Allocate an address from IPAM
  5. Select a node by live free memory
  6. Plan, in this machine's own infrastructure-state workspace
  7. Human approval (60-minute timeout)
  8. Apply - the roll-forward boundary
  9. Register monitoring, minting a unique per-host key into the secrets manager
  10. Configure with Ansible (the agent is already trusted)
  11. Register DNS
  12. Commit the host manifest

Windows runs the same backbone and diverges where it must: it mints a per-host local-admin password into the secrets manager, resolves a template by tag rather than booting a cloud image, and waits for WinRM to be stably up (six consecutive good checks) because first-boot init flaps the port and a single successful probe is a lie.

What broke, three times, and what I did about it.

One: the gate could not see. The collision check asked IPAM for devices and virtual machines. But a host provisioned by this pipeline is recorded in IPAM as an address with a DNS name, not as a device object. So the query came back empty, the API was perfectly healthy, and the gate said "no collision" about a host that was already alive. Fixed by also querying the address records, and by adding a deterministic scan of the configuration inventory: reading a file beats guessing from a naming convention.

Two: the gate could see, and nobody was listening. This one is my favourite. The detector correctly found the collision, printed it, and exited non-zero. The pipeline proceeded anyway. The stage ran the checker piped into tee so the report could be saved, and a shell pipeline exits with the status of its last command. That was tee, which always succeeds. The gate had been reading tee's exit code the entire time it existed. Fixed by redirecting instead of piping, and by adding a fail-safe so an unset or unparseable exit code can never read as clean. The same bug was sitting in the reachability pre-flight; it went in the same pass.

script {
    def rc = (env.COLLISION_RC ?: '').trim()
    if (!rc.isInteger()) {
        // Fail-safe: an unset/unparseable exit code must NEVER read as clean. Gate-or-fail.
        error("Name collision check produced no usable exit code (COLLISION_RC='${rc}'); refusing to proceed ungated.")
    }
    if (rc != '0') {
        // 2 = confirmed collision; 3 = unverifiable (fail-loud). Both gate; wording differs.
        def collision = rc == '2'

The fix for bug two, and the fail-safe it taught me to add.

Three: rejecting the plan leaked the address. Reject the approval and the reserved address stayed reserved. The release ran in the pipeline's failure handler, but a rejected approval does not fail a build, it aborts it. Fixed by moving the release to the handler that covers failed, aborted, and unstable alike, still guarded so that a machine which already exists keeps its address rather than having it yanked out from under it.

Underneath all three is one rule, and it is the part worth stealing: an unreachable system is never "clean". If a system that could hold a conflicting name cannot be reached, that is not a pass, it is an unknown, and an unknown still stops the build.

def evaluate(hits):
    """Pure verdict over the gathered hits. 'collision' > 'error' > 'clean'."""
    if any(h["present"] is True for h in hits):
        return "collision"
    if any(h["present"] is None for h in hits):
        return "error"
    return "clean"


EXIT = {"clean": 0, "collision": 2, "error": 3}

Present means collision. Unknown means error. Only proven-absent is clean.

2. The template factory

The problem. Hand-built golden images rot. They accumulate cruft you cannot see and cannot clean: activation timers that cannot be re-armed again, locked packages, drifted certificates. And you cannot really patch a golden image, you can only add to it. The fix is to stop treating the image as an artifact you maintain and start treating it as an output you regenerate: patching a template means rebuilding it, and the build is code.

  1. Stage - pick the idle half of a live/alternate template pair
  2. Build - Packer, from a custom answer-file ISO
  3. Verify - confirm the thing built is genuinely a template
  4. Promote - tag-swap the new one live, demote the old one
  5. Cleanup on failure - destroy the staging machine, leave the live tag untouched
  6. Email the result

It runs weekly, unattended. Two details carry it. First, the build verifies itself rather than trusting an exit code: it re-reads the machine's config and checks it really is a template, because a stale template left by a previous run must never be misread as this run's success. Second, provisioning never names a template by ID, only by tag, which is what makes the swap invisible to it.

The promote order is the whole trick. Promote the new one to live first, then demote the old one. Do it the other way and a failure between the two steps leaves zero live templates and provisioning is down. This way the worst case is two templates both tagged live, the resolver picks one, and everything keeps working.

def cmd_promote(osname, staging):
    p = PAIRS[osname]; tag = p["tag"]; staging = int(staging)
    if not _is_template(staging):
        sys.stderr.write(f"refusing: staging {staging} is not a template\n"); sys.exit(1)
    old_live = _find_by_tag(tag)
    # Promote staging to live FIRST. If anything fails after this, we have two live
    # templates (resolver picks one - still works) rather than zero (provisioning down).
    node = _node_of(staging)
    _api(f"/nodes/{node}/qemu/{staging}/config", "PUT",
         {"tags": f"win-template;{tag}"})

Fail toward two live templates, never toward none.

What broke: it was finished, and it was not there. This subsystem worked. It had been built, it had run, and the roadmap said "completed". It was also sitting on a branch, more than a hundred and fifty commits behind, entirely absent from the main line, for months. Nobody noticed because everything that consumed it kept working off the old templates. A branch audit found it, and it merged across almost verbatim, needing only four modernizations to meet the standards the main line had since grown. Then a live build ran green from the merged code and the branch was deleted. The lesson was not about Packer: work that is not merged is not done, no matter what the board says.

3. The job reporting pipeline

The problem, in the words I wrote when I filed it: job searching is manual, unstructured, and produces no data. Postings get found ad hoc, fit is judged subjectively, market signal is invisible, and there is no record of which version of a resume went where. So I built the thing I would have built at work: ingest, enrich, score, report, and measure whether the scoring was any good.

  1. Collect (daily) - job boards, aggregators, and a forwarding mailbox, deduped on a URL hash
  2. Enrich (daily) - a local model extracts skills and a 0-10 relevance score
  3. Prefilter - anything scoring low is dropped before a single paid call
  4. Parse salary and deadline deterministically (regex, not a model)
  5. Fit-score the survivors with a paid model, behind a fail-closed budget gate
  6. A human decides - a state machine; choosing "want" triggers resume tailoring
  7. Digest (daily) - quiet by default
  8. Calibrate (weekly) - pure SQL, no model

The routing rule is cheapest-first and it is not a slogan: the free local model reads every posting, the paid model only ever sees what survived it. The digest is quiet by default, which makes its arrival the heartbeat: nothing to report means no email, so an empty inbox is a signal rather than a silence.

The most interesting code in this pipeline is not code. Tailoring a resume with a model is exactly where a system starts lying on your behalf, so the constraint is written as the hardest thing in the prompt:

HARD CONSTRAINT - truthfulness (non-negotiable):
- You MAY reframe wording, reorder sections and bullets, and OMIT anything in the master
  resume that is irrelevant to this posting.
- You MUST NOT add, invent, or imply any employer, job title, employment date, degree,
  certification, metric, or skill that is not already present in the master resume. If the
  posting asks for something the resume does not already evidence, leave it out; do not
  fabricate it. When in doubt, omit rather than invent.
- The KNOWN GAPS listed in the user message are context for what to de-emphasize. They are
  NOT a license to add missing qualifications. Never write experience to fill a gap.

Reframe freely. Never invent. The gaps are not a licence to fill them.

What broke. Three good ones. A hundred and forty files, one database row: the dedupe hash normalized each URL by stripping its query string, which is right for tracking parameters and catastrophically wrong for a source whose posting identity is the query string. Every posting from it collapsed onto a single row, and every run cheerfully re-queued the same work. Fixed with a per-source flag rather than by weakening the rule for everyone. A parser that ate the first letter of every path: the status output was trimmed as a whole, which swallowed the leading column space on the first line only, and then each line was cut at a fixed offset, so the first path lost its first character and the job went looking for a directory one letter short of the real one. A retry with no cap: a posting that consistently failed to score was caught, logged, skipped, and retried again the next run, and the next, forever, burning a budget check each time and hiding the real bug behind a tidy log line. It now counts its attempts, gives up after three, and says so.

Honest about the ending: the calibration machinery is built and tested, but it has not yet had enough real outcomes to judge whether the scores predict anything. It measures nothing until it does, and it will not pretend otherwise. The pipeline also never applies to anything on my behalf: it prepares, a human sends.

4. Self-healing monitoring

The problem. Monitoring that wakes you up to perform the same restart is not monitoring, it is a pager with extra steps. But a system that repairs things unattended is a system that can break things unattended, at machine speed, across a fleet. So the question is not "can it fix this", it is "what is it allowed to do without me".

  1. Detect - classify every broken host. Changes nothing
  2. Email the full plan
  3. Human approval (30-minute timeout; unapproved means zero hosts touched)
  4. Heal Linux
  5. Heal Windows

The engine splits the work by what it can safely own. Drift it can correct at the server (an address that moved, a key that went stale) it fixes directly. Anything needing a touch on the host itself is queued as a plan for a human to approve. Dispatch is OS-aware, and it runs only the roles that repair monitoring, never the provisioning roles that would be destructive on a machine that is already live and serving.

The vocabulary is the honesty. Every run records an outcome per host, and the words it is allowed to use are exactly the words it can prove:

result vocabulary for this engine (Level 1, classify/dispatch only):
  applied    - server-side fix made here (ip-fix / psk-sync)
  dispatched - agent-side heal queued to Ansible (outcome verified later)
  escalated  - no automation path (unknown OS), needs a human
green_within_n / failed are NOT recordable here: this engine does not
re-check the host after healing. That belongs to the post-heal verification
slice (re-query Zabbix availability after the Ansible stages).

It cannot say the heal worked, because it does not look. So it does not say it.

What broke. Per-host monitoring keys live in the secrets manager under two different schemas, for historical reasons. An earlier version of the engine read only one of them. Every host stored under the other schema was silently skipped: not errored, not warned, skipped. And because nothing had failed, the pipeline reported success while those hosts stayed red. That is the worst possible outcome for a self-healing system, worse than not running at all, because a green run is a claim that the fleet is fine. It reads both schemas now.

5. Decommission, with the gate that says no

The problem. Destroying is the one operation you cannot walk back. The machine's name doubles as the selector for its infrastructure-state workspace, which means a typo is not a typo, it is a target. A name matching nothing fails safely and loudly. But a name that happens to match an existing workspace that is empty or wrong does not fail at all, and that is the gap.

  1. Validate input
  2. Plan the destroy
  3. Assert the plan actually destroys something
  4. Human approval (30-minute timeout)
  5. Destroy
  6. Release the address - after the destroy, never before
  7. Deregister monitoring
  8. Deregister DNS
  9. Remove the host manifest, delete the workspace

What broke, and it is the one that bothers me most. A destroy plan that destroys nothing is not an error. It plans zero changes and exits zero. The apply then applies zero changes and exits zero. The pipeline emails "VM Decommissioned". Nothing anywhere had ever asserted that the plan would delete a single thing, because why would a destroy not destroy?

That alone is a lie in an email. What makes it dangerous is the next two stages: deregistering monitoring and DNS keys off the name, not off a confirmed destroy. So a name that matched some abandoned workspace could tear a live host out of monitoring and DNS while destroying nothing and reporting success. And there were more than a dozen junk workspaces lying around from ad-hoc runs, which is exactly what makes an accidental match plausible rather than theoretical.

The fix asserts the plan's contents, not its exit code. It reads the planned changes and looks for an actual delete. Nothing to destroy is exit 2. An unreadable plan is exit 3, and also fails, because a plan you cannot read is not a plan you may act on. It deliberately does not use the tool's own "detailed exit code" flag, which has a documented history of reporting changes when there is nothing to destroy. Best of all it runs before the approval, so a plan that would do nothing never reaches a human to rubber-stamp.

if not deleted:
    print("DESTROY PLAN IS EMPTY: it would destroy 0 resources.", file=sys.stderr)
    print("", file=sys.stderr)
    print("A decommission that destroys nothing is a FALSE NEGATIVE, not a success (item 130).",
          file=sys.stderr)
    print("The plan was built against a workspace whose state has nothing to destroy. Likely:",
          file=sys.stderr)
    print("  - the VM was already destroyed (this decommission has nothing to do), or",
          file=sys.stderr)
    print("  - the wrong workspace was selected, or its state is empty/lost.", file=sys.stderr)
    return EXIT_NOTHING_TO_DESTROY

An empty destroy plan is a false negative, not a success.

Honest about the state of it: this gate is committed and proven against synthetic plans covering every exit path, but it has not yet been re-run live against the two real false negatives that prompted it, so I am not calling it verified. It currently guards the Linux pipeline; the Windows one still carries the older pattern and is next.

6. AIOps morning triage

The problem. Overnight, the estate produces alerts, CI results, and backup outcomes. By morning that is a pile. The pile is not the problem; the problem is that reading it is a job, and the day already has one. So: collect it, have something read it, and put a short honest summary in front of me before I sit down.

  1. Collect from alerting, monitoring, CI, and backups (each writes a heartbeat)
  2. Correlate - deterministically, in code, not asked of a model
  3. Triage - a local model classifies each finding
  4. Escalate the hard cases only - a paid model, twice-braked
  5. Digest - with a health banner

It runs daily at 06:00, after the backups and before me. The heartbeat matters more than it sounds: without it, a collector that never ran and a night with nothing wrong produce the same empty page. A quiet night has to be provably quiet.

Cost gating is two independent brakes, because one is a single point of failure. A fail-closed budget gate that denies when the budget is missing or unreadable, not just when it is exceeded, so a broken secret closes the tap instead of opening it. And a per-run escalation cap on top, which the code calls belt-and-suspenders over the money gate, in as many words:

def should_escalate(local, finding, gate_allowed, escalations_left,
                    conf_threshold=ESCALATE_CONFIDENCE):
    """Decide whether to send this finding to Claude. Returns (escalate: bool, reason: str).

    Never escalates unless the credit gate allows AND a per-run cap remains (belt-and-suspenders
    over the money gate). Given that, escalate a HARD case:
      - local is None (the local model produced unusable output), or
      - local confidence is below threshold (or missing), or
      - high-stakes: severity critical AND locally classified actionable (warrants a 2nd opinion).
    """
    if not gate_allowed or escalations_left <= 0:
        return False, "gate-blocked" if not gate_allowed else "per-run cap reached"
    if local is None:
        return True, "local output invalid"
    _cls, _rat, conf = local
    if conf is None or conf < conf_threshold:
        return True, f"low local confidence ({conf})"

The paid model is for hard cases: bad local output, low confidence, or high stakes.

The one stage allowed to fail the build is the digest. Collectors and triage soft-fail and let the chain continue, because a broken collector should not cost me the whole report. But if the digest cannot send, the build fails loudly, because a report that silently stops arriving looks exactly like a quiet morning. The absence of news has to be news.
